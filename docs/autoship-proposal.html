The controller extracts artifacts from the demo, a human reviews them, tests are generated as the oracle, and a Ralph loop builds the app — one task per iteration, fresh context each time, state via git + progress file.

Problem

The input app usually works just enough to demonstrate value, but it is not a maintainable product.

✗Inline prompts mixed with UI logic
✗Implicit state and hidden coupling
✗No backend contracts
✗No durable schema design
✗No tests
✗No distinction between intended and accidental behavior

The hard part is not "rewrite the code." It's turning a messy demo into reliable artifacts that an implementation loop can converge against.

Product Thesis

vibe-coded demo→ reversed spec→ generated tests→ Ralph loop→ production candidate

The reversed artifacts are the anchor. The generated tests are the oracle. The rewrites are outputs of those artifacts, not direct translations of the original implementation.

What v1 Is & Isn't

What v1 Is

A demo-to-production-candidate rewrite system for LLM-powered workflow apps.

  • AI accounting & reconciliation tools
  • Marketing content generation
  • Internal ops copilots
  • Report generation & document analysis

What v1 Is Not

  • ✗ A guarantee the original demo was correct
  • ✗ A full production-hardening platform
  • ✗ A generic autonomous software factory
  • ✗ A replacement for security/compliance review

"Production candidate", not "fully production-ready."

Key Terms

A

Artifacts

Structured outputs carried between steps. They capture inferred intent, constraints, and assumptions. Artifacts describe.

O

Oracles

The generated tests. They produce pass, fail, or escalation signals. Oracles judge.

L

External-State Convergence Loop

autoship's execution pattern is an instance of a structural pattern that independently emerged in two fields. Ralph loop (Huntley) for coding and autoresearch (Karpathy) for ML research converged on the same structure — evidence the pattern is sound.

Shared structural traits

  1. Fresh context per iteration — no long-running session
  2. Declarative goal state — spec describes the end, not the steps
  3. External persistence — progress file + git, not LLM context
  4. External feedback — machine-verifiable checks, not writer self-judgment
  5. Single-writer, single-process — multi-agent fan-out rejected

The Four Steps

1

Ingest & Reverse Spec

The controller reads the demo repo and produces the artifact set. LLM-powered: extracts product intent, API surface, data model, user flows, design direction, and external dependencies. The reversed spec is a reconstruction, not ground truth.

Task ordering in the PRD: follow the tracer bullet pattern — each slice is an end-to-end functional slice, not backend-only and not full frontend design.

A slice cuts through: route → handler → persistence → tests → minimal UI needed to exercise the path. Enough to prove the path is real (one real screen, one form, stable selectors); not full UI polish.

Phased structure:

  1. Scaffold (single-shot) — shell, nav, design system, routing
  2. Slice 1 — core entity + CRUD + minimal UI, sets conventions
  3. Slices 2–N — one feature each, ordered by data model dependencies
  4. Cross-cutting slice — middleware, shared validation
  5. Polish pass — visual consistency, design system, responsive behavior (separate oracle)

Why this split: backend has one judge (behavioral); frontend has two (functional + visual). Mixing visual quality into a Ralph oracle makes it non-deterministic. Slice loop judges functional correctness; polish pass judges visual quality separately.

2

Human Review

Human Gate

The human validates: who the product is for, core workflows, high-impact actions, data sensitivity, major assumptions. A lightweight acceptance gate, not a full technical review. After review, the artifacts are the contract.

3

Generate Tests

The controller generates the oracle from approved artifacts. Tests are derived from the reversed spec, not the original implementation.

Contract— API shapes, validation, auth, responses
Behavior— business logic fixtures
State— DB writes, session changes, side effects
Journey— Playwright scripts for core user flows

Playwright journey tests are the highest-value user-facing QA signal in the system, but they are noisier than unit or contract checks. Browser-agent walkthroughs and side-by-side UI comparison stay supplementary.

4

Build (Ralph Loop)

Controller spawns headless Claude Code sessions. Each iteration: read artifacts + progress → pick highest-priority task → implement → run the current slice's required feedback loops → block commit unless that slice passes → update progress → exit. Controller validates before next iteration. Full-suite green is reserved for final completion.

Build phases

Pre-loopScaffold (single-shot) — shell, nav, design system, routing. Bounded retry if it fails.
LoopSlice iterations — each slice: backend + minimal UI + journey test. Standard Ralph mechanics.
Post-loopPolish pass — visual consistency, design system, responsive behavior. Different oracle (screenshot regression vs goldens, design lint, a11y).

Frontend oracle — functional vs visual

Backend has one judge (behavioral). Frontend has two targets (functional + visual). Mixing them makes the loop non-deterministic.

Slice loop judges functional correctness: UI exists, right selectors present, journey test passes. Playwright is not perfectly deterministic, but it is still the best end-to-end behavioral oracle for real user journeys, so it stays in the blocking slice gate.

Polish pass judges visual quality: spacing, fonts, responsive variation, with bounded supplementary UI comparison. Good for catching structural regressions (missing components, broken layout). Bad as main oracle — overfits to pixels instead of behavior.

Affordance presence check (per slice)

Before starting the backend loop for a slice, the controller verifies the minimal UI contains every element the journey test references (button labels, form fields, nav targets). Deterministic DOM check. Bounded retry with missing-elements list if it fails. Prevents "backend passes but journey fails at test time because a button is missing."

Complete

Session outputs completion signal. Controller verifies independently.

Iteration cap

Controller enforces configurable limit (5–50). Prevents runaway cost.

Stall

Same task failing across iterations. Controller escalates with specific blocker.

Escalation

Needs spec/test change or human decision. Session exits, doesn't guess.

HITL mode

One iteration at a time. Human reviews between. Recommended for first runs and first tracer bullet slice.

AFK mode

Looped with iteration caps. For bulk work once prompts and artifacts are proven.

progress.txt — not just a checklist. Each entry: completed task + PRD ref, architectural decisions, files modified, blockers, notes for next iteration.

First slice matters most — the agent follows observed patterns over written instructions. If the first slice sets bad patterns, every subsequent iteration amplifies them.

Done = all required tests green across the full suite and no unresolved blocker.

Artifact Model

Six files. Each separate because it serves a different consumer at a different time.

Why separate files? Context management. The controller selects which files each session reads based on the task. A CRUD iteration doesn't need 300 lines of frontend hierarchy. Separate files keep each session's context focused.

Runtime state

Controller protocol lives in code, not in files.

No per-run "dispatch plan" artifact. The controller's source encodes the default protocol — how ingest fans out, how build slices are ordered. Runtime deviations from that default (e.g., "no frontend detected, skipped ui-walker") are appended to decisions.log. To know what the controller intends, read the controller. To know what it did this run, read prompts/ + logs/ + decisions.log.

progress.txt is the single human-readable status file across the whole lifecycle. Its role adapts per step.

  • Ingest / review / generate-tests (bounded synthesis): progress.txt is a derived view. The controller's machine state lives in marker files (phase_*.done). The controller renders progress.txt from state, never parses it to decide what runs next.
  • Build (the true Ralph loop): progress.txt is agent handoff state. Fresh sessions read it each iteration; they write back completed tasks, decisions, and blockers. This is Ralph's original pattern.

Only build is the true Ralph loop. Ingest is a short pipeline; review is a gate; generate-tests is bounded synthesis. One file name for human-readable status is a UX choice, not a claim that all four steps share the same state model.

The Controller

An Agent SDK application — a program that uses LLM calls for judgment and spawns Claude Code sessions for execution.

LLM-powered decisions (controller AI judgment)

  • Extract reversed spec from demo
  • Generate test suite from artifacts
  • Evaluate stalls: code / test / spec problem?
  • Decide: continue / retry / revision step / escalate
  • Independent quality check (independent from the writing session)

Deterministic orchestration

  • Spawn Claude Code sessions with focused context
  • Enforce budgets: tokens, wall clock, iterations
  • Validate progress: check git, run tests independently, don't trust session self-reports
  • Route: continue / retry / escalate

Artifact guardrail: Claude Code sessions should not edit artifacts or tests unless the controller explicitly opens a revision step. If a session hits a problem requiring spec or test changes, it escalates — it does not silently revise the contract it's building against.

Skill assignment: The controller assigns skills to each session based on the task. Backend iteration gets backend-rewrite-loop. Frontend gets frontend-regeneration. Skills are reusable execution playbooks — they carry workflow logic and quality gates.

Executor Mode Options

A natural progression. Each option solves a specific failure mode of the one before it.

A: Can we converge? → B: Can we trust it? → C: Can we go faster? → D: Less human intervention?
v1 — Ship this

A — Single Ralph Loop

"Can we converge?"

One session per iteration. Prove reversed spec + tests = production candidate.

+ Simplest, lowest cost, clear ownership, easiest debugging.

- Can get stuck, no second opinion, may overfit to weak tests.

Next

B — Ralph + Critic

"Can we trust the output?"

A critic session evaluates output — not just "does code match spec" but "is the spec sufficient?" Identifies spec gaps, test coverage holes, wrong assumptions. Proposes improvements to artifacts.

Trigger: A converges but quality is inconsistent — code passes tests but spec itself has gaps.

+ Catches spec gaps, improves artifacts, independent judgment.

- Higher cost, may over-correct, needs revision protocol.

Scale

C — Parallel Slices

"Can we go faster?"

Multiple Ralph loops run in parallel on independent tracer bullet slices. Controller orchestrates dependency ordering and merging.

Trigger: Loop + critic work but too slow for large apps (15+ entities).

+ Faster wall-clock, natural fit with tracer bullets.

- Merge conflicts, shared state assumptions, harder debugging.

Future

D — Self-Correcting Loop

"Can we reduce human intervention?"

When evidence suggests spec/tests are wrong (3 sessions fail same test), controller proposes a revision. Human approves the fix, but controller does the diagnostic work.

Trigger: Loop works but escalates too often.

+ Fewer escalations, automates critic's improvement loop.

- Risk of auto-correcting wrong, requires high trust in controller.

Ship Option A. Earn the rest.

The unproven part is not agent coordination — it's whether the reversed spec and tests are strong enough for the loop to converge.

A fails on quality → B (critic). B is too slow → C (parallel). C escalates too much → D (self-correction). Don't build the next option until the current one proves its failure mode.

Delivery Shape

Two install modes. In-place transforms the demo repo; --from creates a separate workspace.

Default

Mode A — In place

Demo is being replaced by the production candidate.

$ cd my-demo-repo

$ npx autoship init

Moves existing files into prototype/, scaffolds artifacts/, app/, .autoship/. Preserves git history. Reversible.

Flag

Mode B — Separate workspace

Demo stays alive as its own repo for continued prototyping.

$ npx autoship init \

--from ./demo ./new-app

Copies demo into prototype/ as read-only snapshot. Demo repo stays untouched.

Choosing the mode

Prototype is being replaced, nobody iterates on it furtherIn place
Prototype stays alive as a scratchpad--from
Reference-only, historical contextEither

Most common early state: prototype as living scratchpad. Use --from to keep fast-iteration workspace clean.

Daily commands

$ npx autoship ingest # Step 1: extract spec from prototype/

$ npx autoship review # Step 2: human reviews artifacts/

$ npx autoship generate-tests # Step 3: generate oracle tests

$ npx autoship build # Step 4: Ralph loop

$ npx autoship status # progress, blockers

Re-initialization: npx autoship init on an already-initialized workspace fails with an error. Use npx autoship reingest to refresh artifacts from an updated prototype while preserving current app and test work. Separate command, separate intent.

Why This Architecture

1

The user often does not understand the internals of the vibe-coded app well enough to validate a full technical plan.

2

Claude Code is already strong at local code generation and iterative repair.

3

The missing value is not "write code." It's "turn a messy demo into reliable artifacts that make code generation converge."

Considered & Deferred

Ideas explored and cut. Kept to prevent re-derivation.

Trace-diff oracle with corpus splits
Cut: traces freeze stochastic behavior, memorization attacks are trivially gameable, demos encode bugs.
L0/L1 containerize / shell around unchanged logic
Cut: Claude Code already does both in one shot.
Executor mode picker heuristic
Cut: picking between modes is a research project. Ralph alone proves the loop.
Adversarial LLM reviewer
Cut: primarily defended against the now-removed trace oracle.
Production hardening as part of this loop
Cut: productionization applies to any codebase. Separate concern.
Screenshot comparison against demo visuals
Cut: demo UI is usually ugly. Pixel-matching is fidelity to the wrong target.
Agent-browser as the primary loop oracle
Cut: browser-agent judgment is too soft to be the main judge of record. Kept only as supplementary QA at loop time and as synthesis-time help for Playwright generation.
Heavy stage model (DEFINE/PLAN/BUILD/VERIFY/REVIEW/SHIP)
Simplified to 4 steps. The stage model added formalism without changing what the system does. Can be reintroduced from earlier drafts if complex routing is needed later.
3-tier skill model
Simplified. Skills are consumed at session level. The tier taxonomy didn't change how skills are used.
Formal mutable/protected surfaces
Simplified to a guardrail rule: sessions don't edit artifacts or tests unless the controller opens a revision step.
