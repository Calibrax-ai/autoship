Autoship Deliver

A lightweight agent harness for continuous, oracle-guided software delivery

Autoship Deliver is an agent harness that auto-grooms incoming issues, builds the specified change, and verifies it against a runnable oracle — all while staying inside the guard-rails of an existing codebase. The system never lets one agent judge its own work, never drifts scope silently, and never leaves state in memory. Every transition is grounded in code, tests, and reviewer verdicts that persist on disk.

What it is

Agent harness + conventions, not a framework.

Deliver is a new track of the Autoship product family. Where extract turns a messy prototype into a reliable spec, and build drives a full rewrite from that spec, deliver addresses the largest problem surface of all: the everyday work of maintaining and evolving a real, running codebase. Bugs, features, refactors, performance regressions — the unglamorous work that keeps products alive.

Deliver ships as a set of Claude Code agent definitions plus conventions about where state lives. No CLI. No runtime. No build step. You install it by copying agent files into your repo's .claude/agents/ directory; artifacts accumulate under .autoship/issues/<id>/ beside your code. Claude Code is the runtime. The repo is the source of truth. Operators (or, later, a thin controller agent) dispatch the work.

Core claim

The hard part of automated software delivery isn't writing code — LLMs are already strong at that. The hard part is producing trustworthy specifications from fuzzy human intent, and verifying changes against those specifications without any one agent becoming both author and judge. Improve is designed around that insight.

The problem

Existing projects have different failure modes than greenfield builds.

Greenfield systems optimize for discovery — what should this product become? Existing projects optimize for controlled change — this code runs in production, thousands of lines are already deciding behavior, and every change risks breaking something the team forgot existed.

The challenges stack up:

  • The codebase already exists — the agent must respect what is, not invent what could be
  • Issues are narrower than full product specs — one bug, one feature, one refactor at a time
  • Regression risk matters more than discovery breadth — "works + nothing else broke" is the minimum bar
  • Author-equals-judge is devastating here — an agent that writes a fix and confirms "yep, looks good" will ship broken code at scale

These are not hypothetical concerns. Autoship's probe series has run directly into them. Three consecutive probes — 2.2, 2.3, and 2.4 — hit the same structural failure: observe a new failure mode, add a forcing-function gate to catch it, watch the controller cheerfully satisfy the gate while reproducing the same failure under a new label. The accumulated-gates pattern. It took probe 2.5 and a separate fresh-context reviewer agent with a calibration set to break the loop.

The lesson

No amount of rephrasing a self-judged gate fixes self-judgment. The structural fix is to separate the author from the judge. Every handoff in deliver carries a generator-evaluator boundary — the agent that produces never discharges the gates that accept it.

The approach

External state machine, fresh context per unit, evidence anchors every transition.

Autoship Deliver is built around four structural commitments that every probe lesson has reinforced:

1. Evidence-first

Truth comes from code, not from claims

Every brief, every review, every transition cites runnable evidence: a reproduction command, a grep-verifiable file path, a test that actually exercises the change. An agent that claims "root cause is in the auth module" must point to file:line. An agent claiming "alternative A fits our patterns" must cite where the pattern lives.

2. Generator-evaluator

Author and judge are different agents

The pre-groomer writes briefs. The brief-reviewer judges them with fresh context — no memory of how the brief was produced. Same pattern at every later stage: plan-reviewer judges plans, post-build reviewer council judges the built change. The author never marks its own work approved.

3. Fresh context per unit

No session is long-lived

Every structural unit of work — groom, review, regroom, oracle plan, build — runs in a fresh context window. Context accumulation silently degrades agent quality; fresh starts keep the signal-to-noise ratio predictable. State persists on disk between units, not in-session.

4. External state machine

Filesystem is the source of truth

State is derived from filesystem presence: brief.md exists → proposed; reviews/review-NN.md with REJECTED → changes-requested. No parallel state file that can drift from reality. Crash recovery and resumption are both automatic consequences.

End-to-end pipeline

Five stages. Every stage is reviewer-gated. Probe-0.1 validates the first.

At the highest level, deliver is a pipeline that takes an issue from intake to ship. Every stage produces disk-persisted artifacts, and every stage ends with a fresh-context reviewer that either approves the work or sends it back.

The diagram omits the feedback loops for clarity — at every stage, a REJECT verdict loops back to the same stage with updated inputs. Those loops are the generator-evaluator pattern in action: the agent that produced the work is never the one that accepts it. The state machine and per-type grooming diagrams in the next sections show what this looks like in detail.

Probe-0.1 scope

Probe-0.1 validates the Groom stage alone — pre-groom, brief, brief-review, and regroom. It answers one narrow question: can the generator-evaluator pattern produce trustworthy briefs from fuzzy issues? Everything downstream is reserved for probes 0.2+, where grooming's answers shape how Plan, Build, Verify, and Ship are built.

Reading state from disk

No parallel state.json exists. The filesystem is the state machine. This prevents drift between a claimed state and what actually happened, and makes resumption automatic — crash recovery is just "re-read the directory."

Feedback edges

Improve is not a linear chain. At every stage, the reviewer can return REJECT, which loops the work back for another pass:

  • Regroom — brief-reviewer REJECTS → pre-groomer re-grooms from updated inputs. Prior review preserved as review-NN.md history.
  • Re-plan (future) — oracle-reviewer REJECTS → oracle-planner revises.
  • Rebuild (future) — walker or post-build review REJECTS → builder re-dispatches.
  • Rollback + new issue (future) — production monitor detects regression → automatic rollback + investigative issue created.

Issue types

Four types, distinguished by where the truth lives.

Improve recognizes four issue types. The split is not by severity or priority — it's by the shape of grooming each requires. Different types have different truth sources, and grooming has to meet the truth where it actually lives.

Type Grooming shape Truth location Posture
Bug Forensic Latent in code: reproduce → root cause Observe, diagnose, specify
Feature Generative Has to be invented: research → design → pick Smallest design that fits existing patterns
Refactor Structural Code-quality criteria: define "better", preserve behavior Preserve behavior, improve structure
Non-functional Measured In benchmarks: baseline → target Observe a number, not an error
Design discipline

Types are classifications, not pipelines. One pre-groomer + one brief-reviewer handle all four types. Type is a field on the brief; type-specific sections carry the shape differences. Specializing into four grooming pipelines is deferred until observed quality demands it — structure before evidence is formalism.

What's out of scope

Improve explicitly does not handle:

  • Spikes and research tickets — they produce knowledge, not code
  • Strategic planning — "should we rewrite X in Y?" is not a change request
  • Multi-repo coordination — deliver is single-repo by design
  • Standalone documentation updates — doc changes bundled with code are part of the parent type

Bug forensic

The truth is latent in the code — something is broken. Grooming reproduces the bug (confirmed / cannot-reproduce / need-info), traces to a specific file:line, bounds the scope, writes Reproduction Steps and Root Cause sections. Evidence is runtime observation.

Feature generative

The truth has to be designed. Grooming researches existing patterns in the codebase, explores 2-3 alternative approaches with cost + fit tradeoffs, picks the smallest fit. Design Rationale records the alternatives and the pick. Evidence is cited file:line patterns.

Refactor conservational

The truth is: behavior must not change; structure must improve. Grooming inventories existing test coverage (grep, not line-coverage), identifies gaps the refactor must fill first, defines a concrete improvement target with a measurable axis. Behavior Preservation is the load-bearing brief section.

Non-functional measured

The truth is in benchmarks. Grooming measures a baseline, identifies the hotspot, defines a concrete target with tolerance. Same shape as Bug — observation is a number instead of an error. Design deferred to probe-0.2+ once Bug / Feature / Refactor validation completes.

The agent contract

Every dispatch binds four layers: identity, skill, prompt, context.

Every agent invocation in deliver carries four explicit contracts. Making them explicit is how we avoid the default failure mode of "whatever the session drags in" — agents burning turns re-orienting, hallucinating because pre-injection was incomplete, or drifting into files they shouldn't have read.

Rules for extending the system

  • Extend capability with skills. New procedure (reproduce a UI bug, map blast-radius, write a regression test)? Add a skill, not a new agent and not a longer system prompt.
  • Extend orchestration with the controller. Need a different role or fresh context? Add a controller-dispatched step, not a direct agent-to-agent invocation.
  • Never let context be accidental. Every dispatch prompt declares what's pre-injected and what the agent is allowed to discover via tools.
Agents do not invoke agents directly

Consistent with autoship's established pattern, leaf agents never dispatch other leaf agents. If pre-groomer needs information beyond its current task scope, it either invokes a skill in-session (browser automation, DB query, test run) or outputs a structured signal to the controller for a fresh-context dispatch. This preserves single-writer invariants and keeps every transition observable.

The brief schema

Seven base fields, one optional universal, one type-specific section.

The brief is the handoff artifact from grooming to everything downstream. It carries enough structured content to be reviewable by a fresh-context agent, executable by an eventual builder, and diffable when regroom updates it. Base fields are universal; type-specific sections carry what differs.

brief.md │ ├── [Frontmatter] │ ├── issue, issue-rev, groomed-at, trigger, type │ └── type-specific status: │ ├── Bug → reproduction-status │ ├── Feature → design-status │ └── Refactor → preservation-status │ ├── [Base fields — all types] │ ├── Outcome one-line user-visible result │ ├── Acceptance Criteria runnable predicates │ ├── Scope Fence Always / Ask / Never │ ├── Rabbit-Hole Patches pre-answered decisions │ ├── Blast-Radius Manifest Create / Change / May / Must-not │ ├── Skeleton Position single-slice vs multi-slice │ └── Concrete Example input/output or reference │ ├── [Optional universal] │ └── Failure Modes runtime risk scenarios │ └── [Type-specific — exactly one of:] ├── Bug → Reproduction Steps + Root Cause ├── Feature → Design Rationale (menu) │ │ │ ├── Alternatives (always) │ ├── Picked + Reason (always) │ ├── Constraints (conditional) │ ├── Migration Plan (conditional) │ ├── Backward Compat (conditional) │ ├── Rollback Plan (conditional) │ ├── Schema Diff (conditional) │ └── Deferred (optional) ├── Refactor → Behavior Preservation │ │ │ ├── What must be preserved │ ├── Preservation Proof │ └── Structure Improvement └── Non-functional → Baseline + Target (deferred)

Mechanical content (acceptance criteria, blast-radius manifest, subsection presence) is grep-checkable at review time. Judgment content (scope fence, rabbit-hole patches, picked alternative) is what the reviewer actually evaluates. This mapping — mechanical → grep; judgment → reviewer — is a design principle that runs throughout autoship.

Why this schema shape

Each field exists because observation showed briefs without it produce failure:

Field Failure it prevents
Outcome Executors optimizing for AC pass rates instead of user-visible results
Acceptance Criteria "Done" becoming subjective; no runnable check means no verification
Scope Fence Scope drift into adjacent files the change was never meant to touch
Rabbit-Hole Patches Executors silently guessing at ambiguities (and guessing differently each time)
Blast-Radius Manifest "I'll change whatever I need to" — regression by accretion
Skeleton Position First-slice conventions becoming accidental; later slices inheriting bad patterns
Concrete Example Prose interpretation drift — different readers take different meanings

Foundations

Nine load-bearing commitments for probe-0.1.

1 Disk-backed execution state

State lives on disk, not in a long-running session. Crash recovery, resumption, and human steering all fall out for free.

2 Fresh session per unit

Every major unit runs in a fresh context window. Context accumulation silently degrades agent quality — fresh starts keep signal-to-noise predictable.

3 Hard unit sizing

A unit must fit in one context window. If grooming needs more, split into codebase scan, baseline snapshot, and oracle draft as separate units.

4 Explicit state machine

States (new / proposed / changes-requested / ready-for-oracle) are derivable from filesystem presence. No parallel state file to drift.

5 Mechanical gates

The outer loop's transitions are mechanically checkable — brief exists, required sections populated, verdict parseable. Judgment calls live inside the reviewer; the transitions themselves are not judgment-only.

6 Repo-local canonical state

Artifacts live in the target repo (.autoship/issues/<id>/). Trackers (GitHub, Linear) are input and coordination surfaces, not truth sources.

7 Brief schema

Seven base fields, one optional universal, one type-specific section. Each field earns its place by preventing a specific failure mode.

8 Voice-coded anti-pattern framing

Agent prompts carry a short adversarial section — specific failure modes written in a voice the agent internalizes, thicker than a principle list but lighter than a rubric.

9 Pre-injected dispatch context

The dispatch prompt inlines everything the unit needs. The agent is told explicitly what it has and what it does not — no fishing for context the dispatcher already holds.

Anti-patterns

Five patterns deliver explicitly does not adopt.

1. Issue-first truth model

Issue text reflects what the reporter hoped; code reflects what the system does. Autoship's differentiator is evidence-first reasoning — an issue tracker is an input and coordination surface, never the truth source.

2. Replanning as scope escape hatch

Self-judged gates get absorbed by the agent that authored them. Probes 2.2 → 2.4 ran this failure three times, each under a new label. Any replanning must stay reviewer-controlled and evidence-constrained. The reviewer is the fixed point; the plan is negotiable.

3. Ceremony before it pays for itself

No milestone machinery, reporting layers, sync adapters, or recovery subsystems before the first grooming probe proves they are needed. Every additional file is a surface for inconsistency — earn the separation.

4. Static gate registries without calibration

A static list of questions drifts from actual failure modes. The reviewer degrades into a checkbox machine that satisfies its rubric while missing the failure shape that matters. Autoship's calibration methodology — operator overrides become labeled cases; calibration grows over time — is what makes reviewer judgment transferable across probes.

5. Accumulating supervisor modules

When a new failure mode appears, the fix is structural — move the check to the reviewer boundary — not a new supervisor module for every symptom (stuck-detection, timeout-recovery, verification-guard). Each new gate at the author boundary gets absorbed by the author. The reviewer is the fixed point against which structural fixes are made.

Preconditions

Test infrastructure is a precondition for the full pipeline, not for probe-0.1.

Deliver's automation rests on a dependency chain: code change → verification → tests → test infrastructure. Break any link and deliver cannot operate. The full pipeline — grooming through merge — requires:

  • Test infrastructure — runnable test command, functional tests covering observable behavior, deterministic, reasonably fast (minutes not hours)
  • Reproducible environment — seedable database, consistent fixtures, runnable locally or in CI
  • Clean version control — git with a stable main branch to anchor against
Probe-0.1 nuance

These preconditions apply to the full pipeline, not to probe-0.1 specifically. Probe-0.1's grooming stage produces briefs that reference tests; it does not execute them. A testbed with thin tests can host grooming — test infrastructure becomes blocking at oracle and build stages in later probes.

When preconditions are not met, the fix is itself a change. Adding tests, seeding fixtures, or stabilizing CI is a precursor issue that deliver can groom and drive like any other change request.

Probe-0.1 shape

Operator-driven, filesystem-derived state, one happy path.

The first deliver probe stays deliberately narrow. It validates one question: can the generator-evaluator pattern produce trustworthy briefs from fuzzy issues? No code is built. No oracle is planned. No PR is merged. Just brief → review → either approved or sent back for regroom.

0.1-specific simplifications

Each graduates to a richer shape when observed need justifies it — not before.

  • Controller scope stays staged. Current runtime reaches draft PR: claim → pre-groom → review → Ready | needs-human-input, then after human promotion to Building: worktree → Stage 1 → Stage 2 → validation → draft PR. Merge/deploy remain deferred.
  • Skills are inlined in the agent's system prompt. reproduce-api-bug, map-blast-radius, and write-brief live as numbered procedure steps inside pre-groomer.md, not separate SKILL.md files. Promote to skills when a second agent needs them.
  • State derived from filesystem. No state.json; presence of brief.md / reviews/review-NN.md says everything.
  • No calibration set at start. Calibration is born from observed operator overrides, not invented upfront.
  • Reproduction outcome is a brief field. A bug that cannot be reproduced is a brief.md with reproduction-status: cannot-reproduce; reviewer's groundedness check flags it. No separate needs-info.md artifact.
  • Non-functional deferred. Only Bug, Feature, and Refactor are designed for 0.1; Non-functional grooming gets designed when probe-0.2+ has real data.

Post-0.1 extensions

Deferred until observed need justifies the complexity.

Strong architectural choices that don't yet earn their place in probe-0.1. Each waits for a real failure that makes it load-bearing.

1. Branch or worktree isolation

Autonomous work is safer when isolated from the main tree. Becomes load-bearing once deliver builds rather than just grooms.

2. Crash recovery & stuck detection

Detect failed retries, missing artifacts, blocked verification loops. Required before long-running unattended build mode.

3. Stronger verification artifacts

Verification JSON, walker results, reviewer decision records. Supports recovery and learning capture once build lands.

4. Persistent learnings

Closed issues feed back into durable project memory — decisions, learnings, eventually calibration.

5. Tracker adapters

GitHub / Linear integration for coarse issue state sync, review-outcome mirroring, PR opening. Adapters don't define the architecture.

6. Model & complexity routing

Heavy for planning, standard for execution, light for triage. Matters once deliver runs continuously.

7. Multi-reviewer council

At post-build validation: parallel fresh-context reviewers with distinct rubrics (Requirements Coverage, Integration, Acceptance). Any FAIL → remediation.

8. Controller agent for orchestration

At ~10+ issues, the top-level controller running in deliver mode iterates through .autoship/issues/, dispatches sub-agents, and maintains single-writer invariants.

What the architecture delivers

When all foundations are in place and all the deferred extensions have earned their keep, deliver becomes what it was designed to be: a bounded-change delivery engine where evidence anchors every transition, generator and judge are never the same agent, and state lives on disk. Operators move from manually steering individual issues to overseeing a pipeline that handles the unglamorous work at a scale a team of humans alone cannot match — while still refusing to ship a change it cannot defend with evidence.

That is the thesis. Probe-0.1 is the first empirical test of it.
