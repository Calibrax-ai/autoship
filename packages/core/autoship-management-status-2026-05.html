The idea

AI accelerates code. The bottleneck shifts to human attention.

AI agents are fast. But throughput stalls on review, not on the model — a single dev can only supervise so many sessions at once before the cost of switching exceeds the cost of doing the work. The next move isn't a faster coding tool. It's a system where long-running work runs autonomously on a managed hub, and humans manage work, not the sessions producing it.

"The plan is the ceiling." — Autoship learnings, after ten probes
01

Human attention is the bottleneck.

AI agents are fast. A single dev can only supervise a handful at once — not because the model is slow, but because review can't keep up. Throughput stalls on attention, not generation.

02

Long work runs autonomously.

Tasks that take hours run on a managed hub — own workspace, own lifecycle, recoverable across crashes. Not pinned to a developer's laptop, not blocked on human presence.

03

Humans manage work, not sessions.

The team approves intent, reviews PRs, and merges what's right. The system handles routing, execution, and validation. People sit at the merge gate, not the keyboard.

The broader vision

Autoship is a project operating system.

Most agent systems are dispatch layers on top of a single tracker. Autoship's broader bet is that the system itself sits between intent and implementation — taking signals from anywhere, understanding them, prioritizing against the roadmap, and triggering the right work.

01

Gather

Signals from many sources — meetings, monitoring alerts, customer feedback, scheduled research, plus traditional issue trackers.

02

Understand

Raw signals become structured work items through grooming and spec review.

03

Prioritize

Weigh new items against the roadmap and what's already in flight, before anything dispatches.

04

Work

Ready work routes through Groom / Ask / Ship to draft PR with validation evidence.

Today: Gather from Linear, Understand via grooming, Work via the deliver loop. Next: additional Gather sources and cross-stream Prioritize — mapped in the diagram below.

How it works

One agent reads the issue. Specialists do the work.

When intent enters from any spoke, the Controller Agent decides what happens next: groom the plan, ask a human, or hand off to build. Then it dispatches specialists per stage — pre-groomer + spec-reviewer for the plan, oracle-writer + oracle-reviewer for the tests, an implementation agent for the code. Each pair has an author and a judge — no agent grades its own homework.

Blue = agent · amber = human · grey = artifact

Groom

Plan needs work. Controller routes to pre-groomer, which writes a spec; spec-reviewer judges it before any code.

Ask

Missing human input. Autoship posts the specific question in Slack and pauses — no guessing in place of a decision.

Ship

Controller hands off to implementation. A frozen oracle gates the build; the implementer cannot mutate the tests to make them pass.

Today: 6 specialists across 3 states. Roadmap: a structured mini-swarm as Design, Research, and Decomposition states come online — the Controller picks the right specialist for each task.

What it's not good at — yet

Three honest gaps we haven't closed.

All three are also things OpenAI explicitly chose not to solve in Symphony — the closest peer system. Naming them keeps us anchored to the open frontier of the field, not pretending the system is finished.

GAP 01

Planning has no clean entry point.

The path from "I have an idea" to "Autoship can run this" is sharp at the execution end and ambiguous at the planning end. No conversational refinement. No distinct PRD artifact. Today the honest answer to "how do I plan a feature with Autoship?" is "open Claude Code, then write a Linear issue, then run autoship groom" — which means users can rationally stop after step one.

Symphony has the same gap. Its SPEC §2.2 explicitly declares this a non-goal, requiring "agent-ready issue descriptions" as a hard prerequisite. OpenAI dumped the entire intake burden on the user.
Closes with: a planning surface — a plan mode that walks an architect through PRD + decomposition, then hands clean child issues into the existing groom/deliver pipeline.
GAP 02

New projects hit a planning barrier — our biggest adoption hurdle.

Today's strongest results come from codebases that have done their conventionalizing work — tests, CI, project structure, coding style. Autoship reads those signals and operates inside them.

On a new project, the architect still has to pick the stack, package manager, database, auth provider, hosting, and conventions before Autoship has anything to work with. Autoship has no opinions on those decisions and can't make them. That's a planning barrier — and it's the moment most users bounce, because the path of least resistance is to stay in Claude Code or Codex until the project has shape.

Symphony has the same constraint, more explicitly. The much-cited "500% PR increase" comes from OpenAI's own harness-engineered codebase (hermetic tests, reliable CI, agent-ready issues). Critics note Symphony "doesn't deliver the harness you need to make working software" — and neither do we, yet.
Closes with: smarter init that bootstraps standards from a brief description; an opinionated starter mode for greenfield projects.
GAP 03

"Tests pass" isn't "product works."

The frozen oracle gives us a high-confidence signal that the implementation matches the spec. It doesn't tell us the spec was right, or that the running product behaves correctly under real use. We trust the gate. We don't yet trust the outcome.

Symphony has it worse. Symphony has no oracle at all — its stance is "correctness is your CI's problem." We're already ahead here (generator-evaluator, frozen oracle), but the outcome layer is still open for everyone.
Closes with: a runtime/preview verification step beyond the oracle.
Roadmap

From a single delivery loop to a full intent-to-ship system.

The Next column is what we'd build to close the three gaps and light up the intake spokes — but it's gated on what dogfooding tells us. Building more features before proving adoption is wasted leverage. (See Immediate next step.)

Next — close the gaps, light up the spokes

  • Planning surface (plan mode). Conversational PRD intake — turn a rough idea into a PRD and clean child issues without leaving Autoship. Closes gap #1.
  • Greenfield bootstrap. Smarter init that infers standards from a brief description; opinionated starter mode for new projects. Closes gap #2.
  • Outcome verification. A check beyond "tests green" — does the product actually behave? Closes gap #3.
  • Specialist routing across more states. Design, Research, and Decomposition states come online; controller picks the right specialist per task.
  • Multi-source intake. Notion, monitoring alerts, customer feedback — the "Gather" layer from the broader vision.
  • Multi-LLM support. Best (or cheapest) model per task — not bet on one provider.
  • Parallel runs. Multiple issues moving through without blocking.
  • Cross-repo generalization. Second testbed beyond finance_backend_agent.

Later — closing the loop

  • Scheduled research. Bot scans competitors / surfaces ideas on a cadence.
  • Approval gates for sensitive actions. Billing, infra, prod-data — explicit human sign-off.
  • Team-level delivery dashboard. Leadership view of in-flight, blocked, shipped.
  • Learning loop. Shipped work feeds back into how the bot classifies new intake.
Immediate next step

Stop building. Start using.

The architecture has more headroom — planning surfaces, multi-LLM, more specialists. But shipping more features before proving the existing ones get adopted is wasted leverage. Next cycle, the focus shifts from building to dogfooding: using Autoship on our own real work, in the repos we ship from, with the people who ship.

COMMITMENT 01

Run delivery through Autoship on real work, real repos.

Not testbeds. Not synthetic. The issues the team would ship anyway, in the codebases they ship from.

COMMITMENT 02

Track every bounce-out.

Each time a team member abandons Autoship mid-issue and finishes in Claude Code or Codex directly, capture why. Bad grooming? Planning gap? Codebase not conventionalized? The bounces are the highest-signal data we'll get.

COMMITMENT 03

Kill-or-continue checkpoint.

Three honest questions: (a) does the team pick Autoship over a direct AI session? (b) does it work across our real codebases? (c) is the value worth more building? If yes — back to the roadmap with confidence. If no — pause, the work isn't wasted.

More building is the tempting path. Adoption is the load-bearing one. We earn the next feature by using what's built.
